### 1. 现有同类 / 相近产品盘点

#### 1.1 模型仓库 / Model Hub 类（偏“模型托管 + 版本 / 权重”）

- **Hugging Face Hub（Models）**  
  - 定位：**最大开源模型仓库**，包含 LLM 在内的 200w+ 模型。  
  - 特点：模型卡（Model Card）、任务标签、权重、License、评测结果、Spaces Demo。  
  - 不足（相对你的项目）：  
    - 偏“模型托管/开源社区”，**不面向“开发者如何选型 + 场景决策”**，信息偏工程/训练视角。  
    - 中文生态 & 国内闭源商用模型的信息不集中（文心、千问、DeepSeek 等要到各自官网找）。

- **All LLMs Directory / 各类 LLM Models Directory（如 llmmodels.org、LangDB Models）**  
  - 定位：**列出主流 LLM 清单**，按模型家族/版本做列表。  
  - 特点：常见信息有参数量、发布方、开源/闭源、价格、简单描述。  
  - 不足：  
    - 多数是 **“大表格 + 简介”**，缺少深入的**能力维度对比（代码/推理/多模态/长文档等）**和**场景推荐**。  
    - 对国内模型、中文能力、国内价格和接入方式覆盖不全。

#### 1.2 AI 工具 / 模型导航类（偏“工具黄页”）

- **LLM Tools Hub、ExploreAI.tools、AI Data Hub 等 AI 工具导航**  
  - 定位：**全品类 AI 工具导航**，重点是 SaaS 工具，而不是模型本身。  
  - 特点：按场景分类（写作、编程、营销、客服等），每天收录新工具。  
  - 不足：  
    - 模型维度很弱，往往只写“用了 GPT-4/Claude/Gemini”等，**无法做严肃的模型选型对比**。  
    - 对开发者不够“工程化”：缺少 API 细节、SDK 链接、示例代码、调优建议等。

- **中文：AI 工具导航 / 大模型网站排行榜类（如 一流导航、QubitTool 等）**  
  - 定位：**面向普通用户/运营/内容创作者的工具导航**。  
  - 特点：对 DeepSeek、GPT、Gemini、文心、千问、豆包等有简单介绍。  
  - 不足：  
    - 更像“推荐哪几个好用的网站/助手”，**几乎不做系统化参数/能力/价格对比**。  
    - 对“同一家公司多个模型如何选”“不同任务用哪个模型”帮助有限。

#### 1.3 模型评测 / 榜单类（偏“排行榜 + Benchmark”）

- **Hugging Face Open LLM Leaderboard（含 v2 / Model Comparator）**  
  - 定位：**开源 LLM 的标准化评测榜单**。  
  - 特点：多 Benchmark 分数、排名、模型对比工具。  
  - 不足：  
    - 强调的是 **“通用评测分数”**，对开发者实际业务场景（代码生成、知识问答、多轮对话稳定性、接口延迟、成本）没有直观指引。  
    - 覆盖范围主要是开源 LLM，对闭源商用 API（OpenAI/Anthropic/Google/国内云厂商）支持有限。

- **Artificial Analysis 等第三方评测网站**  
  - 定位：独立评测不同模型和 API 性能。  
  - 特点：会有更细的场景测试。  
  - 不足：  
    - 依然是“评测站 + 报告”，**缺少导航/选型知识结构**，需要用户自己解读。

#### 1.4 GitHub Awesome / Model Zoo 类（偏“链接大合集”）

- **LLM-Zoo、awesome-llm、awesome-ai-models 等**  
  - 定位：列出各种 LLM、论文、实现、工具链。  
  - 特点：对研究人员和折腾党很友好。  
  - 不足：  
    - 更偏学术/工程，**不会站在“产品化选型”视角说：这个模型适合谁、为什么、怎么接。**  
    - 很多条目只有链接，没有统一字段结构（参数、价格、能力、场景）。

#### 1.5 中文「大模型 + 知识库 / 学习生态」类

- **AI-Compass（AI 百科）、PandaWiki、k3GPT 等**  
  - 定位：以「AI 全栈学习 / 知识库 / 问答系统」为目标。  
  - 特点：  
    - AI-Compass：覆盖 AI 技术栈，类似“AI 领域的百科 + 教程导航”。  
    - PandaWiki：开源知识库系统，支持大模型助力创作/搜索。  
    - k3GPT：支持接入多种本地/在线大模型做知识共享。  
  - 不足：  
    - 它们是「怎么搭知识库 / 学习 AI」，而不是「帮你选模型」；  
    - 对“不同模型的优缺点、对比和实战用法”也有内容，但**不成体系、非标准化数据结构**。

#### 1.6 模型 API 聚合 / 调度平台（偏“一个 Key 调用多模型”）

- **OpenRouter（国外）、Ofox.ai、各类 Maas 平台、One-API / New-API（自建网关）**  
  - 定位：**通过一个 API 聚合多家大模型**，解决接入与路由问题。  
  - 特点：  
    - 支持数十到数百模型，统一 OpenAI API 兼容接口。  
    - 帮你屏蔽不同厂商的认证、限流、计费细节。  
  - 不足（对你的项目来说反而是机会）：  
    - 这些平台做的是「调用层路由」，**不是“认知层选型 + 知识库”**；  
    - 站内的模型介绍通常非常简略；不会给你一整套「选型指南 + 教程 + 最佳实践」。

### 2. 你这个项目的合理性与差异化

结合上述调研，你的项目 **AI 大模型资源中心（ai-llm-hub）** 的合理性和可差异化方向大致是：

- **定位差异：**  
  - 现有要么做「模型托管 / API 网关」，要么做「工具导航 / 学习百科」，  
  - 你想做的是：**“面向开发者的模型选型导航 + 深度知识库”**，这是一个相对空白的交叉点。

- **用户价值差异：**  
  - 现有站点大多回答的是：  
    - “有哪些模型？”、“模型性能如何？”、“我怎么调用某一家的 API？”  
  - 你希望回答的是：  
    - **“我现在是 X 场景 + Y 约束（预算/延迟/合规），到底该选哪个模型，以及为什么？”**  
    - “选完以后，有没有现成的调用示例、调参建议、最佳实践？”

- **内容结构差异（MVP 非常关键）：**  
  - 不是简单列出模型简介，而是要把模型信息按「统一结构」建模，比如：  
    - 基本信息：厂商、版本、开源/闭源、支持语言、发布日期；  
    - 能力向量：代码、推理、长文本、工具调用、RAG 适配、多模态；  
    - 工程属性：价格（输入/输出）、速率限制、延迟体验、稳定性、SLA、地区可用性；  
    - 场景推荐：适合/不适合的使用场景；  
    - 实战资源：SDK 链接、官方文档、示例项目、踩坑提示。  

  这一层，在调研的产品里，目前都 **不够系统化**，这是你能做出明显价值的核心。

### 3. 建议聚焦的 MVP 范围（先把边界画清楚）

在不做“太多无关功能”的前提下，我建议你的 **MVP 只做下面几件事**：

- **核心对象：模型目录**  
  - 收录：**国内外主流通用 LLM + 少量垂直/代码/多模态模型**（例如 30–80 个“头部 + 典型代表”）。  
  - 为每个模型提供：统一结构的「模型详情页」。

- **核心能力 1：搜索 + 筛选**  
  - 支持按「模型名/厂商」搜索。  
  - 按关键标签筛选：  
    - 语言能力（中/英/多语）、  
    - 典型用途（代码、写作、通用对话、长文档、RAG）、  
    - 模型类型（开源/闭源，本地可部署 / 仅云端 API）、  
    - 价格档位（低/中/高）、  
    - 是否支持多模态、是否有工具调用/函数调用。

- **核心能力 2：模型对比（2~4 个并排）**  
  - 支持勾选 2–4 个模型，**展示一个对比表格**：  
    - 指标维度：能力、价格、上下文长度、是否多模态、是否可本地部署、延迟体验（定性）、常见优缺点。  

- **核心能力 3：场景 → 模型推荐（轻量级）**  
  - 把常见开发场景做成 5–10 个“场景卡片”：  
    - 例如：**代码生成/重构、长文档总结、中文知识问答、RAG 检索增强、Agent/工具调用、多模态问答**。  
  - 点进某个场景，给出：  
    - 适合的几款模型（推荐顺序 + 推荐理由）；  
    - 每个模型的 **“如何快速上手”**（指向官方文档 + 你自己的简要调用示例 / Best Practice）。

- **核心能力 4：知识库骨架（不做复杂交互）**  
  - 几个结构化栏目：  
    - 《如何选择大模型》  
    - 《国内外主流大模型概览》  
    - 《按任务选择模型》  
    - 《成本 vs 效果：如何权衡？》  
  - 初期可以是静态内容页面（Markdown 渲染），不做复杂社区/评论/登录。

- **刻意暂时不做（MVP 排除项）：**  
  - 不做用户登录/收藏/评论/评分。  
  - 不做真正的“API 聚合代理调用”（即不帮用户统一 Key、路由请求）。  
  - 不做复杂个性化推荐 / 智能问答助手（可以预留位置，后续用一个 LLM 接入即可）。  
  - 不做多语言后台/多角色权限管理，先用一个简单的内容管理方式（哪怕先手写 JSON/YAML）。

## 《AI 大模型资源中心（ai-llm-hub）需求规格文档》（MVP）

### 1. 项目概述

#### 1.1 项目名称与 Slogan

- **项目名称**：AI 大模型资源中心（ai-llm-hub）  
- **Slogan**：  
  - 聚合全球大模型，一站式查询与选型  
  - 懂模型，更懂你需要哪一个  
  - 开发者必备的 AI 大模型导航与知识库  
- **网站标题（SEO 友好）**：  
  - AI 大模型资源中心 | 国内外大模型导航、能力对比、开发知识库

#### 1.2 背景与问题

- 开发者在日常工作中需要频繁使用不同大模型完成：需求分析、代码生成、调试、Bug 修复、内容生成等任务。  
- 现状问题：  
  - 信息分散：不同模型的能力、价格、限制散落在各自官网、评测、社区。  
  - 对比困难：很难在一个地方同时对比国内外多家模型的能力、价格、场景适配度。  
  - 选型门槛高：同一厂商多版本模型（如基础版、推理版、代码版、多模态版），缺少清晰的选型建议。  
- 本项目希望通过一个统一的信息架构和内容标准，**降低“模型选型 + 上手”的时间成本**。

#### 1.3 产品定位

- **面向对象**：  
  - 核心：一线开发者（个人开发者、团队工程师）  
  - 次要：产品经理 / 技术负责人  
  - 辅助：AI 爱好者  
- **产品类型**：  
  - 面向开发者的 **大模型资源聚合 + 选型决策辅助平台**  
  - 聚焦“**知道有哪些模型、适合做什么、怎么选、怎么用**”

#### 1.4 MVP 目标

- 提供一个可用的 Web 站点，支持中文用户：  
  - 快速查到国内外主流大模型（约 30–50 个）的核心信息；  
  - 按统一结构对比不同模型的优缺点、价格、能力标签；  
  - 根据典型开发场景，得到推荐模型列表和“如何快速上手”的指引；  
  - 同时具备一套基础的知识库页面，用于系统性科普“如何选择大模型”。

### 4. 功能需求

  - 标注来源；

#### 7.4 可维护性与扩展性

- 数据模型要便于未来扩展：  
  - 后续可增加：代码能力排行榜、IDE 支持度、更多指标。

  - 多语言（中/英）扩展点；

## 3. 技术栈选型与交叉验证

### 3.1 前端技术栈

- 方案 A：**Vue3 + Vite + TypeScript + Pinia + Vue Router**（推荐）
- 方案 B：React + Vite / Next.js

#### 3.1.2 推荐方案：Vue3 技术栈

- **技术组件**
  - 框架：Vue 3（Composition API）
  - 构建工具：Vite
  - 语言：TypeScript
  - 路由：Vue Router 4
  - 状态管理：Pinia（轻量，足够处理对比列表、筛选状态）
  - UI 组件库（可二选一）：
    - Naive UI（现代、对 TS 友好、定制度高）
    - Element Plus（生态成熟，国内熟悉度高）

  - 与你当前仓库 `RuoYi-Vue3` 技术栈高度一致，**学习成本最低**。

  - 如需未来做 SSR/SEO 深度优化，需引入 Nuxt 或自己实现 SSR；
  - 但目前你“不急于对外宣传”，静态站已足够。

> 交叉验证结论：  
> 你已有 Vue3 经验 + 当前项目仓库为 RuoYi-Vue3，选 Vue3 技术栈是 **成本最低 + 风险最小** 的方案，SSR/SEO 不重要时无需增加 Nuxt 复杂度。

> 说明：与 SRS 保持一致的同时，可用 `?` 表示可选字段，方便不同模型信息不完全对齐。

## 6. 部署与运维

### 6.1 部署方式（MVP）

- **构建命令：** `npm run build` / `pnpm build`
- **产物：** `dist/` 静态文件
- **部署目标：**
  - Nginx 静态站点
  - 或任何静态托管（OSS、静态网站空间等）

### 6.2 环境区分

- 初期不必严格划分 dev/prod，可简单：
  - 本地开发：`npm run dev`
  - 生产部署：`npm run build` + 上传 `dist`

### 6.3 日志与监控

- MVP 不需要复杂监控，可采用：
  - 前端简单埋点（如 PV、热门模型点击统计）后续再做
  - Nginx 访问日志即可初步分析

## 7. 安全与扩展性设计

### 7.1 安全

- 主要是静态内容，无用户数据。
- 关注点：
  - 外链跳转时可加 `rel="noopener noreferrer"`。
  - 未来如接入后端，考虑基础的 CORS、鉴权等。

### 7.2 扩展预留

- 将数据访问统一封装在 `services/`，未来可替换为真实 API。
- 数据模型保留扩展字段，方便加入更多指标（如 IDE 支持度、Benchmark 成绩）。
- 路由结构保持清晰，便于未来加入后台页面或管理端。
